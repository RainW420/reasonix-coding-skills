# 失败修复模板

用于把“检查失败 → 诊断 → 最小修复 → 重跑”统一成可审计流程。具体实例可以是 test-fix、build-fix、type-fix 或 lint-fix。

## 核心原则

一次只修一个失败。修失败，不重构。每个结论都来自当前命令输出。

## 流程

1. **检测目标**：确认当前失败来自测试、构建、类型检查、lint 或其他验证命令。
2. **运行命令**：重新运行最小能复现失败的命令，读取完整错误、退出码和相关文件路径。
3. **分组排序**：按根因分组；先修阻断性、上游性、最小范围的问题。
4. **定位根因**：读取报错文件和调用点；不要只凭错误摘要猜测。
5. **最小修复**：只改与当前失败直接相关的代码或配置。
6. **重跑同一检查**：同一命令必须重新运行；失败消失后再处理下一个失败。
7. **停止条件**：同一失败尝试 2 次仍未解决，停止并报告已验证事实、尝试过的修复、下一步需要的上下文。

## 禁止

- 不删除或弱化测试来制造通过。
- 不禁用 lint、类型检查或安全校验。
- 不加无理由的 `any`、空 catch、硬编码或降级依赖绕过问题。
- 不把失败修复升级成架构重构，除非用户明确批准。
- 不在未重跑原失败命令时声称已修复。

## 输出

```
状态：DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT

失败目标：
- 命令：
- 失败摘要：

根因：
- （证据支持的根因）

修复：
- （最小改动）

验证：
- （重跑命令和结果）

剩余问题：
- （如有）
```

## Build / Type / Lint Mode

当构建、编译、类型检查或 lint 失败，需要用最小 diff 修复验证失败时，使用本模式。目标是让指定的 build、typecheck 或 lint 命令恢复通过；不要实现新功能，不做无关重构。

### 工作方式

先遵循 failure-repair 的通用纪律，再按以下目标执行：

1. **识别检查命令**
   - Node: `npm run build`、`pnpm build`、`npm run typecheck`、`npx tsc --noEmit`、`npm run lint`
   - Go: `go test ./...`、`go build ./...`
   - Rust: `cargo test`、`cargo build`
   - Python: `pytest`、`ruff check .`、`pyright .`、`mypy .`
   - 其他项目按现有脚本和 README 选择，不猜命令。

2. **复现失败**
   - 重新运行最小命令。
   - 记录退出码、关键错误、文件路径和行号。

3. **最小修复**
   - 缺导入、类型不匹配、空值检查、配置路径、lint 违规等，只做直接修复。
   - 如果修复需要安装依赖、改架构、改公共 API 或批量重写，停止并报告。

4. **重跑验证**
   - 先重跑原失败命令。
   - 如果原失败通过，再运行相关更高层检查。

### 停止条件

- 同一失败尝试 2 次仍未解决。
- 某次修复引入的新错误多于解决的错误。
- 需要新增依赖或联网安装。
- 需要架构级改动。
- 错误涉及 agent/runner/adapter/protocol 契约但任务未提供契约矩阵。
